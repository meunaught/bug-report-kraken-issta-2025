\section{Introduction}

\textit{KRAKEN: Program-Adaptive Parallel Fuzzing} (ISSTA 2025,
\url{https://dl.acm.org/doi/10.1145/3728882}), presents a new program-adaptive parallel
fuzzer. As part of evaluating KRAKEN's effectiveness, the authors ran it against 37
popular open-source projects and reported the following results:
\begin{itemize}[nosep]
    \item \textbf{192 bugs} found across 37 real-world projects
    \item \textbf{119 CVE IDs} assigned to those bugs
\end{itemize}
These bugs were reported to the developers of the affected software, typically as GitHub
issues or pull requests (but also via Bugzilla, SourceForge, and mailing lists).

\paragraph{Step 1 --- KRAKEN GitHub README.}
The KRAKEN repository at \url{https://github.com/seviezhou/Kraken} contains a Trophy
section in its \texttt{README.md} listing all 119 CVE IDs. These serve as the input for
the CVE-based path.

\paragraph{Step 2 --- CVE JSON lookup (blue path).}
Each CVE ID maps to a JSON record in the \texttt{CVEProject/cvelistV5} GitHub repository.
The path is derived mechanically from the CVE year and numeric prefix:

\begin{center}
\small\renewcommand{\arraystretch}{1.3}
\begin{tabular}{@{}ll@{}}
\toprule
CVE ID   & \texttt{CVE-2021-39550} \\
JSON URL & \href{https://raw.githubusercontent.com/CVEProject/cvelistV5/main/cves/2021/39xxx/CVE-2021-39550.json}{\texttt{.../cves/2021/39xxx/CVE-2021-39550.json}} \\
\midrule
\texttt{references} field & \url{https://github.com/sahaRatul/sela/issues/30} \\
\bottomrule
\end{tabular}
\end{center}

\noindent The \texttt{references} field yields the upstream bug report URL.
This step resolved \textbf{117 out of 119 CVEs} (2 records absent from the database).

\paragraph{Step 3 --- Author-based tracker search (green path).}
The first author, Anshunkang Zhou, operates under different identities across platforms.
Each tracker is queried directly by those identifiers:

\begin{center}
\small\renewcommand{\arraystretch}{1.3}
\begin{tabular}{@{}lll@{}}
\toprule
\textbf{Platform} & \textbf{Identifier} & \textbf{Method} \\
\midrule
GitHub            & \texttt{seviezhou}    & Search API: \texttt{author:seviezhou type:issue} \\
FFmpeg Trac       & \texttt{seviezhou}    & \texttt{/query?reporter=seviezhou\&format=tab} \\
SourceForge       & \texttt{azhouad}      & Per-ticket fetch on giflib, gocr, fig2dev \\
Red Hat Bugzilla  & Anshunkang Zhou       & Known bug IDs confirmed by acknowledgment \\
\bottomrule
\end{tabular}
\end{center}

\paragraph{Step 4 --- Join.}
The two result sets are joined on the bug report URL (after stripping any
\texttt{\#fragment} suffix). A left join from the author-search results onto the CVE
records attaches a CVE ID to each bug where one exists, leaving the field blank otherwise.
The final dataset contains \textbf{191 bug reports} across 32 of 37 KRAKEN projects.

\section{Conclusion}
\label{sec:conclusion}

The pipeline recovered \textbf{191 of the 192 bug reports} across 32 of 37 KRAKEN projects
and confirmed \textbf{105 of the 117 resolvable CVEs} (2 of the 119 KRAKEN CVEs have no
CVEProject record; 12 others appear only in CVE JSON references, not in the author search).

The one-bug shortfall ($192 - 15 + 14 = 191$) reflects two opposing effects:
five projects yielded no results (15 bugs missing), while five others returned more issues
than the paper claims, adding 14 bugs likely filed outside the experiment window.
The 12 unmatched CVEs (\texttt{ncurses}, \texttt{pdftools}) appear in Table~\ref{tab:orphans}.
